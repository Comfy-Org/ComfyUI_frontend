# Coverage-Guided Test Picker

Reads unit and E2E coverage reports, ranks files by coverage gaps weighted by churn, randomly picks one candidate, and generates a self-contained task markdown file that another agent can execute to write the test.

## Phase 0: Locate Coverage Data

Coverage data comes from two sources. Find whichever are available:

### Unit Test Coverage (Vitest)

```bash
# Generated by: pnpm test:coverage
# Output: coverage/lcov.info (LCOV format)
UNIT_LCOV="coverage/lcov.info"
```

### E2E Coverage (Playwright + monocart)

```bash
# Generated by: pnpm test:browser:coverage (or CI workflow ci-tests-e2e-coverage.yaml)
# Output: coverage/playwright/coverage.lcov
E2E_LCOV="coverage/playwright/coverage.lcov"
```

### Fallback: Download from CI

If local coverage files don't exist, download from the latest CI run:

```bash
# Find the latest successful run on main
RUN_ID=$(gh run list --workflow="ci-tests-e2e-coverage.yaml" --branch=main --status=success --limit=1 --json databaseId -q '.[0].databaseId')
gh run download "$RUN_ID" --name e2e-coverage --dir temp/coverage/

# For unit coverage
RUN_ID=$(gh run list --workflow="ci-tests-unit.yaml" --branch=main --status=success --limit=1 --json databaseId -q '.[0].databaseId')
gh run download "$RUN_ID" --name unit-coverage --dir temp/coverage/
```

If neither local nor CI coverage exists, tell the user and stop.

## Phase 1: Parse & Rank

### 1a. Parse LCOV

For each LCOV file, extract per-file stats. Use the project's existing parser as reference (`scripts/coverage-report.ts`) or parse directly:

```bash
# Extract file, lines found (LF), lines hit (LH) from LCOV
grep -E '^(SF:|LF:|LH:)' "$LCOV_FILE" | paste - - - | \
  awk -F'[:\t]' '{file=$2; lf=$4; lh=$6; missed=lf-lh; pct=(lf>0?lh/lf*100:100); print missed, pct, lf, file}' | \
  sort -rn
```

### 1b. Filter Out Noise

Remove from candidates:
- Files at **100% line coverage** (nothing to test)
- Files at **0 lines** (empty/generated)
- Files matching exclusion patterns: `*.d.ts`, `locales/**`, `assets/**`, `*.test.ts`, `*.spec.ts`, `*.stories.ts`
- `src/lib/litegraph/**` (excluded from vitest coverage config, has its own test suite)

### 1c. Weight by Git Churn

Files changed more often are higher-value test targets. Compute a composite score:

```bash
# Get commit count per file in last 90 days
git log --since="90 days ago" --name-only --pretty=format: -- 'src/**' | \
  grep -v '^$' | sort | uniq -c | sort -rn > /tmp/churn.txt
```

**Composite score** = `missed_lines × (1 + log2(churn_count + 1))`

This ensures high-churn files with coverage gaps float to the top.

### 1d. Classify Candidates

For each candidate, determine test type:
- **Unit test** if the file is a utility, composable, store, schema, or pure function (check path: `composables/`, `stores/`, `utils/`, `services/`, `schemas/`)
- **E2E test** if the file is a Vue component, view, or UI-heavy module (check path: `components/`, `views/`, or `.vue` extension)
- **Either** if ambiguous — note both options in the task file

### 1e. Select Top 20

Take the top 20 by composite score. Print the ranked list for the user.

## Phase 2: Random Pick

Select one candidate randomly:

```bash
# Pick random index from 0-19
PICK=$((RANDOM % 20))
```

Show the user which file was picked and why it ranked (missed lines, churn, coverage %).

## Phase 3: Generate Task File

Create a markdown task file at `temp/plans/test-task-<filename>.md` with all context needed for a naive agent to write the test.

### 3a. Discover Related Skills

Survey both the repo and global skills directories for test-related skills. Do NOT hardcode skill names — discover dynamically:

```bash
# Repo skills
find .claude/skills/ -name "SKILL.md" -exec grep -li \
  -e "test" -e "playwright" -e "e2e" -e "browser" -e "vitest" \
  -e "unit test" -e "component test" -e "flak" -e "coverage" {} \;

# Global skills
find ~/.claude/skills/ -name "SKILL.md" -exec grep -li \
  -e "test" -e "playwright" -e "e2e" -e "browser" -e "vitest" \
  -e "unit test" -e "component test" -e "flak" -e "coverage" {} \;
```

For each match, read the `name:` and `description:` from frontmatter. Include in the task file as `use-skill: <name>` directives only for skills whose description is relevant to the picked test type (unit vs e2e).

### 3b. Discover Related Documentation

Find all markdown files in `browser_tests/` at any nesting level:

```bash
find browser_tests/ -name "*.md" -type f
```

Also include:
- `docs/testing/*.md`
- `docs/guidance/playwright.md` (if e2e)
- `docs/guidance/vitest.md` (if unit)

### 3c. Determine Test Placement

Find the existing test file or module where this new test belongs:

- **Unit tests**: Check if `<source-file>.test.ts` already exists next to the source. If yes, add tests there. If no, create colocated.
- **E2E tests**: Search `browser_tests/tests/` for spec files that test the same feature area. Use directory structure and file names to find the best fit. If no match, suggest the most appropriate subdirectory.

Include the path of the target test file and a brief note about why it fits.

### 3d. Audit Available Helpers & Patterns

Scan and list reusable infrastructure the agent should use:

**For E2E tests:**
```bash
# Fixtures
ls browser_tests/fixtures/*.ts browser_tests/fixtures/components/*.ts \
   browser_tests/fixtures/helpers/*.ts browser_tests/fixtures/utils/*.ts

# Test helpers
ls browser_tests/helpers/*.ts browser_tests/utils/*.ts

# Types
ls browser_tests/types/*.ts
```

For each file found, read exports (grep for `export`) and include a one-line summary of what it provides.

**For unit tests:**
- Check `src/` for shared test utilities (e.g., `__tests__/`, test helpers)
- Note relevant vitest patterns from `docs/testing/unit-testing.md`

### 3e. Assemble the Task File

The task file MUST contain these sections:

```markdown
# Test Task: <filename>

## Target File
- **Source**: `<path>`
- **Current coverage**: <X>% lines, <Y> missed lines
- **Churn score**: <N> commits in last 90 days
- **Test type**: unit | e2e

## Coverage Gaps
<List the specific uncovered lines/functions/branches if parseable from LCOV>

## Skills to Load
<Dynamically discovered use-skill directives>

## Reference Documentation
<List of all relevant .md files with brief description of each>

## Test Placement
- **Target test file**: `<path>`
- **Rationale**: <why this file/location>

## Project Guidelines
- Colocated tests: `*.test.ts` next to source (unit), `browser_tests/tests/*.spec.ts` (e2e)
- Frameworks: Vitest + @testing-library/vue (unit), Playwright (e2e)
- Run: `pnpm test:unit -- <path>` or `pnpm test:browser:local -- --grep "<name>"`
- Lint after: `pnpm typecheck && pnpm lint`
- Tags (e2e): @smoke, @slow, @screenshot, @canvas, @node, @widget, @mobile, @2x
- No `waitForTimeout` in e2e — use retrying assertions
- No change-detector tests — test behavior, not implementation

## Available Helpers & Patterns
<Audited list of fixtures, helpers, utils, page objects, and what they export>

## Anti-Patterns to Avoid
- Don't test mocks — ensure tests fail when real code breaks
- Don't mock what you don't own
- Don't create new utility files if existing helpers cover the need
- Don't add a new spec file if an existing one covers this feature area
```

## Phase 4: Spawn Work Environment

### Check for tmux + worktree-utils

```bash
# Check tmux
TMUX_AVAILABLE=false
if [ -n "$TMUX" ] && command -v tmux &>/dev/null; then
  TMUX_AVAILABLE=true
fi

# Check worktree-utils
WT_AVAILABLE=false
if command -v wt-new &>/dev/null; then
  WT_AVAILABLE=true
fi
```

### If both available

1. Create worktree: `wt-new ComfyUI_frontend test/<branch-name>`
2. Move task file into the worktree: `mv temp/plans/test-task-*.md <worktree-path>/temp/plans/`
3. Spawn tmux window: `tmux new-window -c <worktree-path> -n "test-<filename>"`
4. Done — tell user "Task file ready in new tmux window."

### If not available

Print what's missing and how to install:

```
Missing: wt-new (git-worktree-utils)
  → brew install git-worktree-utils && git-worktree-utils-setup
  → Manages bare-repo + worktree layout for parallel feature branches

Missing: tmux
  → brew install tmux (or apt install tmux)
  → Terminal multiplexer for running parallel agent sessions

Manual steps:
  1. git worktree add ../test-<branch> -b test/<branch>
  2. cp temp/plans/test-task-*.md ../test-<branch>/temp/plans/
  3. cd ../test-<branch> && <start your agent>
```

## Anti-Patterns

- ❌ **Hardcoding skill names** — Always discover dynamically from `.claude/skills/` directories
- ❌ **Picking the #1 ranked file** — Random selection from top 20 ensures breadth
- ❌ **Generating the test itself** — This skill generates the *task file*; a separate agent writes the test
- ❌ **Ignoring existing test files** — Always check for existing coverage before suggesting new files
- ❌ **Running without coverage data** — Don't guess; require actual LCOV data
